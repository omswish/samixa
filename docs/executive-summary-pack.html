What the product delivers

  • One-screen engineering dashboard for HCI, HSD, network, and servers.
  • Separate operator and admin web surfaces using the same deployed application stack.
  • Live section freshness, last-sync visibility, and source-attribution rules for trustable operations.
  • Admin controls for service restart, source configuration, and collector session recovery.

Key operating principle

The dashboard must not fabricate values during normal operation. If a collector fails, the system keeps showing the last successfully synced data together with the last synced time for the affected card or section.

Data Trust Model

Source-of-truth rules and freshness behavior

Operationally critical

Source rules

  • Nutanix is the source of truth for HCI-backed servers.
  • SolarWinds is the source of truth for network telemetry and on-prem server telemetry.
  • For overlapping server coverage, Nutanix remains primary unless it is stale for more than 10 minutes and SolarWinds telemetry exists.

Freshness rules

  • Each section stores last attempt, last success, poll interval, and last error.
  • Cards show the last known good data when collectors fail.
  • The UI uses section health states instead of synthetic fallback values.

What operators can trust

  • Displayed values come from real collector output, not fabricated placeholders.
  • Data-link status tells whether the values are current, stale, errored, or never synced.
  • Fallback state is explicit when Nutanix-backed servers temporarily show SolarWinds values.

Implication for review

This is a status-preserving operational dashboard, not a predictive analytics surface. Review and audit discussions should focus on freshness, source attribution, and recovery controls rather than whether missing data is synthesized.

Security Position

Pragmatic internal deployment with compensating controls

Needs documented controls

Controls already present

  • Separate operator and admin surfaces.
  • Signed role cookies and admin-only API guards.
  • Loopback-only runtime config and secret endpoints.
  • Optional encrypted secret storage in PostgreSQL.

Known limitations

  • Local password-only auth today.
  • Browser session files required for SolarWinds and HSD.
  • Internal HTTP still exists inside the runtime model.
  • Nutanix TLS verification is relaxed due to source certificate issues.

Compensating controls

  • Deploy on a dedicated internal Windows server or VM.
  • Expose only ports 21060 and 21061 to the LAN.
  • Restrict access by firewall and network ACL.
  • Lock runtime folders to admins only.

Recommended near-term hardening

  • Move production secrets into PostgreSQL encrypted storage.
  • Rotate `APP_AUTH_SECRET` and the shared local password at deployment time.
  • Introduce stronger application-layer authentication for administrative writes.
  • Put the front doors behind internal TLS termination when feasible.

Audit posture

The application is defensible for an internal ISMS review when positioned as a controlled, LAN-hosted operational dashboard on a dedicated server with documented compensating controls and operational procedures.
