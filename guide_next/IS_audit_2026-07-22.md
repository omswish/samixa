# Information Security Audit

Date: 2026-07-22  
Project: Utkal IT Dashboard  
Branch reviewed: `dev-omkars`

## Scope

This review covers the current application and deployment code in:

- `api-gateway/`
- `dashboard/`
- `frontdoor-proxy/`
- `collectors/`
- `deployment/`
- PostgreSQL migration assets in `api-gateway/sql/` and `api-gateway/src/postgres.ts`

This is a code-and-deployment audit, not a live penetration test, AD/SSO review, or infrastructure hardening review of the target Windows server, firewall, switch ACLs, or reverse proxy.

## Executive Summary

The application has improved from its earlier state in three important ways:

- collector secrets can already be encrypted at rest instead of stored only as plaintext
- runtime passwords changed from the admin UI are stored as salted hashes instead of plaintext
- PostgreSQL audit structures already exist for collector config, secret changes, app users, ingest events, status history, and snapshots

However, the current build still has multiple material ISMS gaps. The highest-risk issues are:

1. plaintext source credentials and default app passwords still written into `.env`
2. unauthenticated loopback endpoints that expose decrypted runtime secrets and allow collector reconfiguration
3. unauthenticated collector ingest on `/api/update`
4. admin and operator surfaces exposed to LAN over plaintext HTTP with `secure: false` cookies
5. remote admin sessions can trigger server-local remediation helpers because the backend does not enforce locality
6. PostgreSQL TLS verification is disabled when SSL is enabled

In its current form, the application is usable for controlled internal operations, but it is not yet ready to pass a strict corporate ISMS review without compensating controls and code changes.

## Current Security Positives

- Collector secrets are encrypted with AES-256-GCM before local/PostgreSQL storage in `api-gateway/src/secretCrypto.ts`.
- Collector configuration and secret audit tables already exist in PostgreSQL:
  - `api-gateway/sql/005_postgres_collector_target_config_audit.sql`
  - `api-gateway/sql/006_postgres_collector_secret_config.sql`
- PostgreSQL also already supports:
  - ingest event history
  - collector run history
  - dashboard state snapshots
  - asset status history
  - app user and app user audit tables
- Runtime app passwords changed through the admin UI are hashed with `scrypt` in `dashboard/src/lib/local-auth.ts`.
- Some sensitive internal endpoints are restricted to loopback in `api-gateway/src/index.ts`:
  - `/api/runtime-config/:source`
  - `/api/runtime-secrets/:source`
  - `/api/admin/settings`

These are useful foundations, but they do not yet close the main audit gaps listed below.

## Findings

### F-01 High: Plaintext collector credentials and app passwords are still persisted in `.env`

## Evidence

- `.env.example`
- `deployment/installer/support/provision-staged-deployment.ps1`
- `deployment/installer/utkal-it-dashboard.iss`
- `ecosystem.config.js`

The deployment flow writes the following into the installed application `.env`:

- `NUTANIX_PASS`
- `SW_SERVERS_PASS`
- `SW_NETWORKS_PASS`
- `SYM_PASS`
- `APP_ADMIN_PASSWORD=17172737`
- `APP_OPERATOR_PASSWORD=17172737`
- `APP_LOGIN_PASSWORD=17172737`

## Risk

- anyone with filesystem access to the host can recover source credentials
- default app passwords are predictable and identical across installs unless manually changed
- a server backup, support archive, or accidental file copy leaks production secrets directly

## Required Fix

- treat `.env` as bootstrap-only, not steady-state secret storage
- after first successful bootstrap, move source credentials out of `.env`
- remove app password values from `.env` entirely once runtime auth storage is active
- generate per-install random initial passwords or require explicit setup input
- never ship `17172737` as a default in installer, provisioner, or runtime fallback

## ISMS Impact

Fails basic secret management and default credential expectations.

### F-02 High: Decrypted runtime secrets are exposed on unauthenticated loopback endpoints

## Evidence

- `api-gateway/src/index.ts:459`
- `api-gateway/src/runtimeSecrets.ts`

`GET /api/runtime-secrets/:source` returns decrypted usernames and passwords to any loopback caller. It is not authenticated; it is only source-restricted by client address.

## Risk

- any local process, local user session, malicious browser helper, or host compromise can extract all collector credentials
- loopback-only is not sufficient as a trust boundary on shared or admin-accessible Windows hosts

## Required Fix

- require a machine-local authentication control for collectors and admin services
- preferred options:
  - named pipe or Windows service ACL boundary
  - mutual TLS between gateway and collectors
  - per-service bootstrap token stored outside the web app surface
- do not expose decrypted secrets over a generic HTTP route

## ISMS Impact

This is a major secret exposure weakness even if the gateway itself is not LAN-exposed.

### F-03 High: Collector configuration endpoints are mutable from loopback without authentication

## Evidence

- `api-gateway/src/index.ts:479`
- `api-gateway/src/index.ts:491`

`GET /api/admin/settings` and `PUT /api/admin/settings` rely only on loopback restriction. The dashboard adds admin auth before calling them, but the gateway itself does not.

## Risk

- any local process can change collector URLs, owners, polling intervals, usernames, and passwords
- an attacker with low-friction local execution can silently redirect data collection to hostile endpoints or poison the configuration state

## Required Fix

- enforce authz on the gateway-side admin settings endpoints as well
- require a signed internal service token or a Windows-local authenticated channel
- log all configuration reads and writes with actor identity

## ISMS Impact

Current control is insufficient for privileged configuration management.

### F-04 High: `/api/update` collector ingest is unauthenticated

## Evidence

- `api-gateway/src/index.ts:374`
- `api-gateway/src/index.ts:23`

The gateway accepts collector payloads without any authentication, signature, or source binding.

## Risk

- if the gateway is ever exposed beyond loopback, anyone can inject false telemetry or failure states
- even on loopback, any local process can forge collector updates
- current ingest event logging does not prevent tampering; it only records after the fact

## Required Fix

- require collector-to-gateway authentication
- minimum acceptable control:
  - per-collector HMAC signing with rotated shared secrets
- better control:
  - mTLS between collectors and gateway
- reject unsigned or stale payloads
- include nonce/timestamp replay protection

## ISMS Impact

Weakens data integrity, which is central to this dashboard’s purpose.

### F-05 High: Admin and operator portals are exposed to LAN over plaintext HTTP

## Evidence

- `frontdoor-proxy/src/index.ts:6`
- `ecosystem.config.js:72`
- `ecosystem.config.js:85`
- `deployment/config/service-manifest.json`
- `deployment/installer/support/configure-firewall.ps1`

The front doors bind to `0.0.0.0`, are intended for LAN access, and use plain HTTP.

## Risk

- credentials and session cookies can be intercepted on the local network
- traffic can be modified in transit
- admin console exposure over LAN materially increases blast radius

## Required Fix

- terminate HTTPS in front of the application
- if HTTPS cannot be implemented immediately, restrict the admin port to loopback or a tight IP allowlist
- keep operator and admin on separate exposure policies
- document temporary compensating controls if plain HTTP must remain

## ISMS Impact

Plaintext administrative access is a significant audit concern.

### F-06 High: Session cookies are deliberately non-secure

## Evidence

- `dashboard/src/app/api/auth/login/route.ts:32-33`
- `dashboard/src/app/api/auth/logout/route.ts:16-17`

Cookies are set with:

- `httpOnly: true`
- `sameSite: 'lax'`
- `secure: false`

## Risk

- cookies travel over HTTP
- cookies can be stolen by any observer on the network path
- `sameSite=lax` helps somewhat against CSRF but does not solve transport insecurity

## Required Fix

- set `secure: true` when served behind HTTPS
- add strict origin checks on admin POST routes
- add session rotation on privilege elevation and password change

## ISMS Impact

Not acceptable for production admin access without compensating controls.

### F-07 High: Hardcoded default app passwords remain active in code and installer flows

## Evidence

- `.env.example:32-34`
- `ecosystem.config.js:57-59`
- `deployment/installer/utkal-it-dashboard.iss:260-262`
- `deployment/installer/support/provision-staged-deployment.ps1:364-366`
- `dashboard/src/lib/local-auth.ts:60-63`

## Risk

- predictable credentials across environments
- easy password reuse between old and new installs
- likely failure in audit interviews and control walkthroughs

## Required Fix

- remove all password fallbacks from code and installer defaults
- force setup-time password entry or random one-time bootstrap generation
- require immediate password change on first admin login

## ISMS Impact

Default credentials are typically non-compliant unless strictly temporary and enforced at first run.

### F-08 High: Remote admin can trigger local remediation helpers because backend locality is not enforced

## Evidence

- `dashboard/src/lib/server-auth.ts`
- `dashboard/src/app/admin/page.tsx` hides some actions based on `session.isServerLocal`
- `dashboard/src/app/api/admin/sessions/launch-reauth/route.ts`
- `dashboard/src/lib/admin-runtime.ts:1023-1117`

The UI checks locality for presentation, but the backend route does not reject non-local admin sessions before launching local PowerShell/Edge helpers.

## Risk

- remote admin users can trigger local browser helper windows on the server
- can disrupt operations or interfere with session-dependent collectors

## Required Fix

- enforce `session.isServerLocal === true` inside the route, not just in the UI
- apply the same rule to other host-local remediation and service-control actions if intended

## ISMS Impact

Backend authorization does not match intended operational policy.

### F-09 High: PostgreSQL SSL mode disables certificate validation

## Evidence

- `api-gateway/src/postgres.ts:170-171`

When `POSTGRES_SSL` is enabled, the code uses:

- `ssl: { rejectUnauthorized: false }`

## Risk

- TLS is present but server identity is not verified
- enables man-in-the-middle risk on database connections
- can invalidate the value of moving secrets and audit data into PostgreSQL

## Required Fix

- require certificate validation in production
- trust the correct CA or pin the server certificate
- keep insecure SSL only for explicitly flagged development mode

## ISMS Impact

This is a critical issue if PostgreSQL becomes the security system of record.

### F-10 High: Nutanix TLS verification is disabled globally in the collector

## Evidence

- `collectors/nutanix/src/index.ts:10`

The collector sets `NODE_TLS_REJECT_UNAUTHORIZED = '0'`.

## Risk

- disables certificate validation for every HTTPS request made by that process
- allows MITM against Nutanix telemetry collection
- weakens confidence in authoritative HCI metrics

## Required Fix

- import the internal CA or pin the Nutanix certificate
- if impossible immediately, isolate the path network-wise and document the compensating control
- scope any bypass to the specific target if temporary bypass must remain

## ISMS Impact

Known exception may be acceptable only with strong documented compensating controls.

### F-11 High: Shared local roles (`admin` and `operator`) provide weak accountability

## Evidence

- `dashboard/src/lib/local-auth.ts`
- login is tied to fixed identities `admin` and `operator`

## Risk

- individual accountability is weak
- audit cannot attribute actions to named persons
- shared admin credentials are usually challenged during ISMS review

## Required Fix

- move to named users with role assignments
- ideally integrate SSO/AD or a controlled user directory
- if local fallback remains, store named accounts and log each admin action

## ISMS Impact

Weak non-repudiation and poor privileged-user accountability.

### F-12 Medium: No login throttling, lockout, MFA, or password policy enforcement

## Evidence

- `dashboard/src/app/api/auth/login/route.ts`
- `dashboard/src/app/api/admin/app-auth/route.ts`

Current password update validation only enforces minimum length 4. There is no attempt throttling or lockout.

## Risk

- brute-force risk on LAN-exposed login endpoints
- weak passwords can be set
- admin access is single-factor only

## Required Fix

- add rate limiting and login attempt cooldowns
- enforce minimum password length and complexity
- if possible, adopt SSO/MFA for admin users

## ISMS Impact

Fails common password and access control expectations.

### F-13 Medium: Long-lived stateless sessions cannot be centrally revoked

## Evidence

- `dashboard/src/lib/auth-cookie.ts`
- `dashboard/src/app/api/auth/login/route.ts`

Operator sessions last 365 days by default. Session tokens are signed but not server-stored; there is no session registry or revocation list.

## Risk

- password changes do not automatically invalidate existing sessions
- stolen cookies remain valid until expiry
- no immediate kill switch for individual sessions

## Required Fix

- shorten default session lifetime
- maintain session state server-side or use token versioning
- invalidate existing sessions on password change and high-risk admin actions

## ISMS Impact

Weak session lifecycle control.

### F-14 Medium: Admin actions and local-mode config changes lack a trustworthy audit trail

## Evidence

- local auth writes `app-auth.json` directly in `dashboard/src/lib/local-auth.ts`
- local collector settings are written in `api-gateway/src/localCollectorStore.ts`
- service actions and reauth launches are not persisted as structured audit records

## Risk

- impossible to prove who changed what in SQLite/local mode
- forensic reconstruction is weak
- operational misuse may be invisible

## Required Fix

- persist admin actions to an append-only audit trail
- include actor, timestamp, source IP, action, target, old value, new value, result
- if PostgreSQL is adopted, use it as the audit system of record

## ISMS Impact

Auditability is materially incomplete in current local mode.

## PostgreSQL Migration Assessment for ISMS

## Short answer

Yes, PostgreSQL can materially improve the application’s ISMS posture, but only if it becomes part of the actual security control plane, not just an optional mirror.

## What PostgreSQL already gives you

The current codebase already has the beginnings of a strong audit model in PostgreSQL:

- dashboard state snapshots
- asset current state
- asset telemetry history
- gateway ingest event logs
- collector run history
- asset status history
- collector target config audit
- collector secret config audit
- app user and app user audit tables

This is substantially better than SQLite/local JSON for:

- traceability
- current-state normalization for dashboard reads
- historical metric storage for trends and graphs
- centralized audit review
- retention governance
- backup and restore
- controlled read/write permissions

## What PostgreSQL does **not** solve by itself

Moving to PostgreSQL alone will **not** fix the main audit gaps unless the following are also changed:

1. plaintext bootstrap secrets in `.env` and installer flows
2. loopback unauthenticated runtime secret/config endpoints
3. HTTP-only admin/operator exposure
4. local shared-account authentication
5. disabled TLS verification for PostgreSQL itself
6. unencrypted session-cookie storage JSON on disk

## Current PostgreSQL limitations in this codebase

- PostgreSQL is optional, not primary
- application login is still local-file based
- collector secrets are encrypted before storage, but the master passphrase still lives in env
- PostgreSQL SSL validation is disabled
- there is no demonstrated database role separation in the application layer

## Recommended PostgreSQL target state

To leverage PostgreSQL for ISMS properly, move to this model:

### Phase 1: Make PostgreSQL the primary control store

- collector target configuration
- collector secret metadata and ciphertext
- app users and role assignments
- admin action audit logs
- login success/failure records
- asset current state used by the live dashboard
- asset telemetry history used by trend and graph views
- asset status history used by uptime, outage, and SLA-style reporting

### Phase 1A: Make PostgreSQL the primary metrics store

- use `asset_current_state` as the authoritative current-state read model for live cards
- use `asset_telemetry_history` as the authoritative raw time-series store for observed metrics
- use `asset_status_history` as the authoritative state transition history for availability/status reporting
- preserve provenance per point with `truth_source`, `fallback_source`, `status_origin`, and `quality`
- keep SQLite/local JSON only as standalone fallback or temporary bootstrap mode, not as the long-term graph source

### Phase 1B: Add metric retention and rollup policy

- keep raw metric points for a short operational window such as 30 to 90 days
- add hourly rollups for medium-term trend analysis
- add daily rollups for long-term wallboard trends and management reporting
- define purge and archival jobs so historical growth remains predictable and auditable
- document which cards need raw near-real-time points versus aggregated history

### Phase 2: Retire local secret dependence

- use `.env` only for bootstrap and platform wiring
- remove source passwords from the steady-state `.env`
- store ciphertext in PostgreSQL
- protect the master encryption key outside the app data path
  - Windows DPAPI
  - certificate-backed secret
  - enterprise secret vault

### Phase 3: Enforce proper database transport and access control

- enable PostgreSQL TLS with certificate validation
- use a dedicated least-privilege application role
- separate read/write/admin roles if operations require them
- ensure backups are encrypted and access-controlled

### Phase 4: Use PostgreSQL for actual accountability

- named users instead of shared `admin` and `operator`
- login event table
- admin action table
- password change history or delegated SSO

## Validation conclusion

PostgreSQL is a good fit for ISMS needs **if** the project commits to:

- making PostgreSQL primary for configuration and audit
- making PostgreSQL primary for current and historical metric data
- removing plaintext secret dependence from `.env`
- fixing TLS validation
- wiring live auth and audit to the database or SSO

If PostgreSQL is adopted only as a mirror while the app continues using local auth, loopback secret endpoints, and plaintext `.env` secrets, the ISMS benefit will be limited.

## Recommended Remediation Order

### Immediate

1. Remove hardcoded default passwords and force per-install password setup.
2. Lock down `/api/runtime-secrets`, `/api/admin/settings`, and `/api/update` with real service authentication.
3. Enforce server-local checks in backend routes for reauth/remediation helpers.
4. Stop exposing the admin surface broadly to LAN, or apply IP restrictions immediately.
5. Fix PostgreSQL TLS validation and document the Nutanix certificate exception separately.

### Near-term

1. Move steady-state collector credentials out of `.env`.
2. Integrate named-user auth and audit, preferably via PostgreSQL or SSO.
3. Add login throttling, password policy, and session revocation.
4. Add HTTPS in front of both portals and mark cookies `secure`.
5. Switch dashboard graph/history reads to PostgreSQL instead of local mirrored state.

### Medium-term

1. Make PostgreSQL the primary audit/config store.
2. Add retention, archival, rollup, and backup controls for metric history.
3. Remove debug tooling from production payloads.
4. Address `npm audit` dependency findings with tested upgrades.

## Bottom-Line Readiness

Current state:

- acceptable for tightly controlled internal pilot use with documented compensating controls
- not yet suitable for clean ISMS approval as a production operational platform

PostgreSQL decision:

- recommended, but only as part of a broader security hardening step
- not sufficient on its own
