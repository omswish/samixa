# Information Security Audit
Date: 2026-07-14

Scope: `samixa` wallboard application and supporting collectors as present in the workspace on 2026-07-14.  
Constraint followed: no application code or config changes were made as part of this audit; this file is documentation only.

## 1. Executive Summary

The application is functionally progressing, but it is **not yet ready for a corporate ISMS / internal security audit pass** in its current form.

The main blockers are:

1. **Credential and session handling is unsafe**.
2. **The gateway and dashboard are exposed without authentication or transport protection**.
3. **Collectors can write high-value browser session tokens and debug artifacts into the workspace**.
4. **The system lacks input validation, auditability, and hardening expected for enterprise operations**.

## 3. Detailed Findings

### F-01. Live-looking credentials and internal endpoints are present in `.env.example`
Severity: **Critical**

Evidence:

- `.env.example:2-15`

Why this is a problem:

- The example file contains what appears to be real usernames, internal IPs, and a shared password pattern rather than placeholders.
- Even if the password is already rotated, this is a high-confidence audit finding because it signals weak secret hygiene and poor separation between sample config and operational secrets.
- The same credential pattern is shown for Nutanix, SolarWinds, and Symphony, which also suggests **credential reuse across systems**.

Practical fix / mitigation:

1. Immediately rotate all credentials that ever matched this file.
2. Replace `.env.example` with placeholders only, for example `your_username`, `your_password`, `your_host`.
3. Move all production secrets into a managed secret store such as CyberArk, HashiCorp Vault, Azure Key Vault, or an equivalent enterprise tool.
4. Enforce separate service accounts per system:
   - `svc_itdash_nutanix`
   - `svc_itdash_solarwinds`
   - `svc_itdash_symphony`
5. Document ownership, rotation frequency, and approval flow for each secret.

Audit expectation:

- No real or realistic secrets in tracked examples.
- No shared credential reuse across unrelated platforms.

### F-02. Gateway and dashboard are intentionally bound to all interfaces and exposed without authentication
Severity: **Critical**

Evidence:

- `ecosystem.config.js:7-10`
- `ecosystem.config.js:37-41`
- `api-gateway/src/index.ts:11-19`
- `api-gateway/src/index.ts:71-108`
- `api-gateway/src/index.ts:110-111`
- `dashboard/src/app/page.tsx:2595-2635`

Why this is a problem:

- The API gateway listens on `0.0.0.0`.
- The Next.js dashboard is started with `-H 0.0.0.0`.
- The dashboard automatically connects to `ws://<current-host>:4000`.
- There is no user authentication, no service authentication, no IP allowlist, and no origin validation.

Impact:

- Any reachable machine on the same network can:
  - read operational data from `/api/status`
  - subscribe to live WebSocket updates
  - attempt to post forged updates to `/api/update`
- This is a direct **confidentiality and integrity** risk.

Practical fix / mitigation:

1. Put both dashboard and gateway behind a reverse proxy with:
   - HTTPS/TLS
   - corporate SSO or AD-based authentication
   - source IP allowlisting for collector endpoints
2. Do not expose the gateway directly on the LAN unless there is a documented business requirement.
3. Restrict the gateway to `127.0.0.1` if it is only meant for local collector traffic.
4. Split public read access and internal write access onto separate listeners or separate services.
5. Add WebSocket origin checks and authenticated upgrade handling.

Audit expectation:

- No unauthenticated internal state API exposed to the subnet.

### F-03. `/api/update` accepts unauthenticated writes and has no schema validation
Severity: **Critical**

Evidence:

- `api-gateway/src/index.ts:80-107`
- `collectors/nutanix/src/index.ts:63-73`
- `collectors/solarwinds/src/index.ts:236-245`
- `collectors/symphony/src/index.ts:47-57`

Why this is a problem:

- Any client that can reach the gateway can submit update payloads.
- The gateway trusts caller identity entirely based on JSON shape.
- There is no JSON schema validation, no signature, no replay protection, and no rate limiting.

Impact:

- An attacker or misconfigured host can inject false data into executive-visible dashboards.
- False "all green" or false "all down" states are possible.
- This undermines operational trust and incident response.

Practical fix / mitigation:

1. Add **collector authentication**:
   - mTLS between collector and gateway, or
   - signed JWT with short expiry and collector identity claim, or
   - HMAC-signed payloads with rotating shared keys
2. Add strict schema validation with `zod`, `ajv`, or equivalent before processing writes.
3. Reject payloads containing multiple source objects unless explicitly intended.
4. Add request size limits, rate limiting, and audit logging of caller identity.
5. Record source host, source identity, and result in an append-only audit table.

Audit expectation:

- Writes must be attributable, authenticated, validated, and logged.

### F-04. Nutanix TLS verification is globally disabled
Severity: **High**

Evidence:

- `collectors/nutanix/src/index.ts:7-8`

Why this is a problem:

- `NODE_TLS_REJECT_UNAUTHORIZED = '0'` disables certificate verification for Node HTTPS requests in that process.
- This enables man-in-the-middle interception against the Nutanix collector path.

Impact:

- Nutanix credentials can be captured.
- Cluster telemetry can be tampered with in transit.

Practical fix / mitigation:

1. Remove the global TLS bypass.
2. Import the proper internal CA certificate into the runtime trust store.
3. If required, pin the Nutanix endpoint certificate or CA chain.
4. Fail closed on certificate validation errors and surface the collector state as stale/error.

Audit expectation:

- No production data collector should disable TLS verification globally.

### F-05. Sensitive browser sessions and tokens are stored in workspace runtime folders
Severity: **High**

Evidence:

- `collectors/solarwinds/src/sessionPaths.ts:3-7`
- `collectors/symphony/src/sessionPaths.ts:3-6`
- `collectors/solarwinds/src/index.ts:464-475`
- `collectors/solarwinds/src/index.ts:999-1020`
- `collectors/symphony/src/index.ts:145-155`
- `collectors/symphony/src/index.ts:268`
- Observed local runtime artifacts under:
  - `collectors/solarwinds/runtime/solarwinds-servers-storage-state.json`
  - `collectors/solarwinds/runtime/solarwinds-networks-storage-state.json`
  - `collectors/solarwinds/runtime/solarwinds-servers-profile/`
  - `collectors/solarwinds/runtime/solarwinds-networks-profile/`
  - `collectors/symphony/runtime/`

Why this is a problem:

- Storage-state JSON and persistent browser profiles contain cookies, local storage, session state, and potentially federated identity tokens.
- These are placed inside the project workspace.
- If the workstation or repository folder is copied, backed up insecurely, or indexed, sessions can be replayed.

Impact:

- Session hijack risk
- Lateral movement into SolarWinds / Symphony
- Identity token leakage

Practical fix / mitigation:

1. Move runtime/session storage outside the repo tree into a locked OS-managed path, for example:
   - Windows: `%PROGRAMDATA%\\UAIL\\itdash\\sessions`
2. Apply filesystem ACLs so only the service account and administrators can read/write.
3. Encrypt sensitive runtime/session files at rest.
4. Add TTL-based cleanup and explicit logout/invalidations where possible.
5. Do not store persistent browser profiles unless absolutely necessary; prefer short-lived storage-state only.

Audit expectation:

- High-value session material must not live in an ordinary working directory.

### F-06. Debug artifact capture can write authenticated HTML/screenshots to disk
Severity: **High**

Evidence:

- `collectors/symphony/src/index.ts:124-143`
- `collectors/symphony/src/index.ts:625-627`
- `collectors/solarwinds/src/debug_page_source.ts`
- `collectors/solarwinds/src/debug_widget_content.ts`
- `collectors/solarwinds/src/debug_net.ts`
- `collectors/solarwinds/src/debug_status.ts`
- `collectors/solarwinds/src/debug_net_api.ts`

Why this is a problem:

- Debug runs and optional debug mode can save full page HTML and screenshots from authenticated business systems.
- These artifacts may include ticket titles, categories, user names, internal topology, session state, and other sensitive operational data.

Practical fix / mitigation:

1. Hard-disable debug artifact generation in production mode.
2. Move debug output to a secured temp location outside the repo.
3. Auto-delete debug artifacts after a short retention window.
4. Redact session tokens, user identifiers, and business data before saving any support artifacts.
5. Gate debug mode behind an explicit environment flag and document approval for its use.

Audit expectation:

- Production scrapers should not silently accumulate authenticated page dumps on disk.

### F-07. Internal application traffic still uses plaintext HTTP
Severity: **High**

Evidence:

- `ecosystem.config.js:17`
- `ecosystem.config.js:24`
- `ecosystem.config.js:31`
- `collectors/nutanix/src/index.ts:10`
- `collectors/solarwinds/src/index.ts:9`
- `collectors/symphony/src/index.ts:10`
- `collectors/solarwinds/src/login.ts:19`
- `collectors/solarwinds/src/login.ts:26`
- `collectors/solarwinds/src/index.ts:845`

Why this is a problem:

- Collector-to-gateway traffic uses `http://localhost:4000/api/update`.
- SolarWinds access is over `http://10.36.91.45` and `http://10.36.91.46`.
- If any of this traffic leaves loopback or traverses a switched LAN segment, it is vulnerable to interception and tampering.

Practical fix / mitigation:

1. Serve the gateway over HTTPS, even internally, if it is exposed beyond loopback.
2. Prefer collector-to-gateway loopback only, or a local named pipe / Unix socket equivalent if feasible.
3. For SolarWinds, enable HTTPS on Orion if operationally possible.
4. If the upstream system cannot support TLS, isolate the traffic at the network level and document compensating controls.

Audit expectation:

- Operational telemetry and sessioned scraping should not rely on plaintext transport without a documented exception.

### F-08. Permissive CORS and no WebSocket origin/auth checks
Severity: **High**

Evidence:

- `api-gateway/src/index.ts:11`
- `api-gateway/src/index.ts:19-48`

Why this is a problem:

- `app.use(cors())` accepts cross-origin browser requests by default.
- WebSocket connections are accepted with no origin or identity validation.

Impact:

- Browser-based abuse from other origins becomes easier if a user can reach the gateway.
- Unauthorized dashboards or scripts can subscribe to live updates.

Practical fix / mitigation:

1. Replace open CORS with a strict allowlist.
2. Validate `Origin` on WebSocket upgrade.
3. Require an authenticated session or signed token before returning data.
4. Separate machine-writer endpoints from human-reader endpoints.

### F-09. No security headers / CSP hardening on the frontend
Severity: **Medium**

Evidence:

- `dashboard/next.config.js:1-6`

Why this is a problem:

- There is no CSP, HSTS, X-Frame-Options, Permissions-Policy, or Referrer-Policy defined in app config.
- Even if a reverse proxy will eventually add these, the application currently expresses no baseline hardening intent.

Practical fix / mitigation:

1. Add response headers either in `next.config.js` or, preferably, at the reverse proxy:
   - `Content-Security-Policy`
   - `X-Frame-Options`
   - `Referrer-Policy`
   - `Permissions-Policy`
   - `X-Content-Type-Options`
   - `Strict-Transport-Security` when HTTPS is in place
2. Document the final authority for security headers if they are proxy-managed.

### F-10. Full operational state is returned to any dashboard client, including internal metadata
Severity: **Medium**

Evidence:

- `api-gateway/src/index.ts:71-74`
- `api-gateway/src/index.ts:27-28`
- `api-gateway/src/db.ts:14-81`
- `dashboard/src/app/page.tsx:2612-2617`

Why this is a problem:

- The state model includes internal hostnames, IPs, node IDs, last boot times, machine types, hardware types, and other asset metadata.
- Even if the UI hides some fields, the API and WebSocket still distribute the full object graph.

Practical fix / mitigation:

1. Create a **public dashboard projection** containing only fields required for display.
2. Keep administrative or diagnostic metadata on a restricted endpoint.
3. Add role-based data projection if multiple audiences are expected.

Audit expectation:

- Data minimization should apply to operational dashboards.

### F-11. No append-only audit trail for state changes or collector writes
Severity: **Medium**

Evidence:

- `api-gateway/src/db.ts:287-302`

Why this is a problem:

- The database stores one mutable `app_state` row.
- This is good for runtime, but weak for forensics, accountability, and ISMS evidence.
- There is no immutable history of who wrote what, when, from where, and whether it was accepted.

Practical fix / mitigation:

1. Add an `audit_events` table with:
   - timestamp
   - collector identity
   - source IP / host
   - section updated
   - validation result
   - checksum / request ID
2. Retain audit records per corporate policy.
3. Add tamper-evident logging or forward logs to SIEM.

Audit expectation:

- Operational state changes should be attributable and reviewable.

### F-12. Legacy JSON migration leaves plaintext backup artifacts
Severity: **Medium**

Evidence:

- `api-gateway/src/db.ts:323-327`

Why this is a problem:

- JSON state is copied to `*.legacy-json.bak` before removal.

### F-13. Verbose logging includes internal URLs and raw scraped business metrics
Severity: **Medium**

Evidence:

- `collectors/solarwinds/src/index.ts:983-1054`
- `collectors/symphony/src/index.ts:247`
- `collectors/symphony/src/index.ts:659-660`

Why this is a problem:

- Logs reveal infrastructure endpoints and scrape behavior.
- Symphony currently logs the full scraped payload via `JSON.stringify`.

Practical fix / mitigation:

1. Replace raw payload logging with structured, redacted summaries.
2. Avoid logging URLs, hostnames, and detailed error bodies unless necessary.
3. Apply retention and access control to runtime logs.

### F-16. Service account separation / least privilege is not evident
Severity: **High**

Evidence:

- `.env.example:8-15`

Why this is a problem:

- SolarWinds and Symphony are shown with the same user identity pattern.
- Least privilege, blast-radius control, and clean offboarding are weakened when service access is shared.

3. Document owner, approver, expiry, and rotation process.

## 4. Positive Observations

These are not enough for audit pass, but they are worth keeping:

1. `.env` is ignored (`.gitignore:3`), so secrets are not intentionally tracked in Git.
2. Collector runtime folders for SolarWinds and Symphony are at least partially gitignored (`.gitignore:7-8`).
3. The gateway tracks collector health and last-success timestamps, which is a useful control for operational truthfulness.
4. The application does not appear to use obvious dangerous browser sinks like `dangerouslySetInnerHTML`.

## 5. Required Remediation Before ISMS Review

These are the minimum actions I would classify as **must complete before presenting this application to corporate IS / ISMS reviewers**:

1. Rotate all credentials and sanitize `.env.example`.
2. Move session/token artifacts out of the repo tree and purge existing runtime/session files.
3. Put gateway and dashboard behind authenticated HTTPS access.
4. Add authenticated, validated collector-to-gateway writes.
5. Remove Nutanix TLS bypass and implement trusted CA validation.
6. Disable or tightly control debug artifact generation in production.

## 6. Recommended Remediation Order

### Phase 1: Immediate containment

1. Rotate all potentially exposed credentials.
2. Purge all existing storage-state JSON, browser profiles, screenshots, HTML dumps, and cached runtime artifacts from operator machines.
3. Restrict gateway exposure to loopback or a tightly controlled reverse proxy.

### Phase 2: Identity and transport hardening

1. Introduce SSO / AD auth for viewers.
2. Introduce collector authentication for `/api/update`.
3. Enforce TLS end-to-end where supported.

### Phase 3: Secure engineering controls

1. Add schema validation, rate limits, audit trail, and logging redaction.
2. Add CSP and other browser security headers.
3. Add CI security gates:
   - SCA
   - secret scanning
   - lint rule for forbidden debug artifact writes in production paths

### Phase 4: Audit-readiness

1. Document architecture, trust boundaries, service accounts, and data flow.
2. Define retention for logs, database, debug artifacts, and sessions.
3. Record break-glass and credential rotation procedures.

## 7. Final Assessment

Current audit posture: **Fail**

Reason:

- The application currently exposes live operational data without authentication.
- It allows unauthenticated writes to the state gateway.
- It stores sensitive authenticated browser session material on disk in the workspace.
- It disables TLS verification for one collector.
- It contains poor secret hygiene in tracked example configuration.

If the remediation in Sections 5 and 6 is implemented, the application can move from "audit fail" to "reviewable with compensating controls". Without those changes, it is unlikely to pass a corporate ISMS or internal information security review.
